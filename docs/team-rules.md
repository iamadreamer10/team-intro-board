# 팀 규칙




## 1. 이슈(Issue) 규칙
모든 작업의 시작은 이슈 생성부터 진행합니다.

* **이슈 기반 작업**: 모든 변경 사항(문서 수정, 코드 추가 등)은 반드시 GitHub 이슈가 선행되어야 합니다.
* **이슈 생성 양식**:
    * **제목**: `[번호] 작업 내용 짧은 요약` (예: `[#1] 팀원 소개 문서 작성`)
    * **내용**: 해당 이슈에서 달성해야 할 체크리스트를 포함합니다.
* **이슈 번호 활용**: 생성된 이슈 번호(예: #4)는 브랜치명과 커밋 메시지에 반드시 포함합니다.

---

## 2. 브랜치(Branch) 전략
모든 작업은 독립된 브랜치에서 진행하며, `main` 브랜치의 안정성을 유지합니다.

* **기본 원칙**: `main` 브랜치 직접 Push 금지
* **작업 브랜치**: 모든 작업은 `feature` 브랜치에서 진행하며, 작업 완료 후 PR을 통해서만 `main`에 반영합니다.
* **주요 브랜치 명명 규칙**:
    * `#2` 메인 페이지: `feat/readme`
    * `#3` 팀 규칙: `feat/rules`
    * `#4` 팀원 소개: `feat/intro-이름`
    * `#5` 추천 주제: `feat/topics`
    * `#6` 수정 이력: `feat/changelog`

---

## 3. 커밋 컨벤션(Commit Convention)
이력을 명확히 관리하기 위해 아래 형식을 준수합니다.

* **커밋 메시지 형식**: `<타입> : [이슈 번호] <제목>`
    * *예시: `feat: [#2] 팀 규칙 설정`*
* **커밋 타입**:
    * **feat**: 새로운 기능 추가
    * **docs**: 문서 수정 (README, 규칙 등)
    * **fix**: 버그 수정
    * **refactor**: 코드 리팩토링 (기능 변화 없음)
    * **style**: 코드 의미에 영향을 주지 않는 변경사항 (포맷팅 등)
    * **test**: 테스트 코드 추가
    * **chore**: 빌드 설정, 패키지 매니저 수정 등 기타 작업
---

## 4. PR(Pull Request) 전략
팀원 간의 교차 검증을 통해 코드/문서의 질을 높입니다.

1.  **Main Push 금지**: 모든 작업 브랜치에서 `main`으로 갈 때는 반드시 PR을 생성합니다.
2.  **동기화**: `pull` 대신 `fetch` 후 로컬에서 직접 `merge` 또는 `rebase` 여부를 판단하여 동기화합니다.
3.  **최소 1명 리뷰**: PR 머지 전 최소 1명 이상의 팀원 리뷰를 받아야 최종 병합이 가능합니다.
4.  **머지 후 관리**: 작업이 완료된 feature 브랜치는 삭제하여 목록을 청결하게 유지합니다.
